# Project Management System

##  Описание проекта

Система управления проектами с трехуровневой архитектурой, реализующая:
- CRUD операции для проектов, сотрудников и задач
- Фильтрацию и сортировку данных
- Аутентификацию и авторизацию с ролями (Director, ProjectManager, Employee)
- Drag & drop загрузку документов
- REST API с Swagger документацией

## Роли и права доступа

Система реализует трехуровневую модель авторизации, основанную на ролях: Director (руководитель), ProjectManager (менеджер проекта) и Employee (сотрудник). Каждая роль обладает строго определенным набором прав, которые распространяются на все CRUD-операции с сущностями системы.

**Director** обладает полным и неограниченным доступом ко всем ресурсам системы. Ему разрешены операции чтения, создания, обновления и удаления для всех сущностей: сотрудников, проектов и задач. Это единственная роль, которая может создавать новых сотрудников, изменять их данные или удалять их из системы, а также удалять проекты.

**ProjectManager** имеет расширенные права на управление проектами и задачами, но ограничен в управлении кадровым составом. Менеджер проекта может просматривать любые проекты и задачи, создавать новые проекты и задачи, обновлять их параметры, а также управлять составом исполнителей в рамках своих проектов. Однако ProjectManager не может создавать, изменять или удалять сотрудников, а также удалять проекты — эти операции остаются исключительной прерогативой Director.

**Employee** обладает наименьшим набором прав, ограниченным преимущественно чтением. Сотрудник может просматривать список проектов и задач, получать детальную информацию по конкретным объектам, а также искать сотрудников. Единственное действие, доступное сотруднику помимо чтения, — изменение статуса задач, что позволяет ему отмечать выполнение поставленных задач. Сотрудник не может создавать, изменять или удалять проекты, задачи или сотрудников.

Такая модель разграничения доступа обеспечивает четкое разделение ответственности: руководители управляют кадровым составом, менеджеры проектов координируют работу и распределяют задачи, а исполнители фокусируются на выполнении поставленных задач и информировании о завершении работ.

## Аутентификация

Система использует JWT (JSON Web Token) для аутентификации. Для регистрации нового пользователя необходимо выполнить POST-запрос на endpoint `/api/Auth/register` с передачей в теле запроса объекта, содержащего поля `firstName`, `lastName`, `patronymic` (опционально), `email`, `password` и `role` (значения: Director, ProjectManager или Employee). При успешной регистрации в ответ возвращается JWT токен, а также данные пользователя и его роли. Для входа в систему используется POST-запрос на `/api/Auth/login` с передачей `email` и `password`. В ответ также возвращается JWT токен, который необходимо включать в заголовок `Authorization: Bearer {token}` при всех последующих запросах к защищенным ресурсам.

## Управление сотрудниками

API для работы с сотрудниками доступно только авторизованным пользователям. Получение списка всех сотрудников выполняется GET-запросом на `/api/Employees`. Для получения конкретного сотрудника по идентификатору используется GET `/api/Employees/{id}`. Поиск сотрудников по имени, фамилии или email осуществляется через GET `/api/Employees/search` с передачей query-параметра `term`. Создание нового сотрудника доступно только пользователям с ролью Director и выполняется POST-запросом на `/api/Employees` с телом запроса, содержащим `firstName`, `lastName`, `patronymic` и `email`. Обновление данных сотрудника также доступно только Director через PUT `/api/Employees/{id}`. Удаление сотрудника выполняется DELETE-запросом на тот же endpoint и также требует прав Director.

## Управление проектами

Работа с проектами реализована через endpoints группы `/api/Projects`. Получение списка проектов с поддержкой фильтрации и сортировки выполняется GET-запросом, который принимает query-параметры: `StartDateFrom` и `StartDateTo` для фильтрации по дате начала, `Priority` для фильтрации по приоритету, а также `SortBy` (поддерживаемые значения: name, startdate, priority) и `SortDescending` для управления порядком сортировки. Получение конкретного проекта по идентификатору осуществляется GET `/api/Projects/{id}`. Создание нового проекта доступно Director и ProjectManager через POST-запрос, требующий передачи объекта с полями `name`, `customerCompany`, `executorCompany`, `startDate`, `endDate`, `priority`, `projectManagerId` и списком `executorIds`. Обновление проекта выполняется PUT-запросом с аналогичной структурой данных. Удаление проекта разрешено только Director и выполняется DELETE-запросом. Управление исполнителями проекта реализовано через два дополнительных endpoint: POST `/api/Projects/{projectId}/executors/{employeeId}` для добавления исполнителя и DELETE `/api/Projects/{projectId}/executors/{employeeId}` для удаления. Оба метода доступны Director и ProjectManager.

## Управление задачами

Задачи являются частью проектной иерархии и управляются через endpoints `/api/Tasks`. Получение списка задач поддерживает фильтрацию по статусу (ToDo, InProgress, Done), приоритету, идентификатору проекта и исполнителю через соответствующие query-параметры: `Status`, `Priority`, `ProjectId`, `ExecutorId`. Сортировка возможна по полям `title`, `priority`, `status`, `createdAt`. Получение конкретной задачи осуществляется GET `/api/Tasks/{id}`. Создание задачи через POST-запрос требует передачи `title`, `comment` (опционально), `priority`, `status`, `projectId`, `authorId` и опционального `executorId`. Обновление задачи выполняется PUT-запросом. Для изменения статуса задачи предусмотрен отдельный PATCH-метод `/api/Tasks/{id}/status`, принимающий в теле запроса строковое значение нового статуса. Также реализованы специализированные GET-методы для получения задач по проекту (`/api/Tasks/project/{projectId}`) и по исполнителю (`/api/Tasks/executor/{executorId}`). Доступ к задачам имеют все авторизованные пользователи, но создание, обновление и удаление доступны только Director и ProjectManager. Изменение статуса задачи доступно всем ролям.
